# Organizations Design

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Add multi-member organization support to AI Native Core, migrating billing and token budgets from per-user to per-org, with email invites, shareable invite links, role-based access, and an org switcher in the UI.

**Scope:** Phase 17 items 84 (org settings), 85 (invite flow), 86 (org roles), 87 (org billing), 88 (org switcher). Items 101–102 (security alerts, mobile push) remain deferred.

---

## Architecture Overview

Every user gets a personal organization auto-created on signup. The existing `tenants` table IS that personal org — `org_id = user_id` for all existing rows, so backward compatibility is preserved. We extend `tenants` in-place (no rename) and add two new tables. Billing, token budgets, and notifications all migrate from `user_id` to `org_id` via the auth context.

**Tech Stack additions:** None — uses existing Stripe, Resend, asyncpg, Next.js, shadcn/ui.

## Data Model

### Columns added to `tenants`

```sql
ALTER TABLE tenants ADD COLUMN slug TEXT UNIQUE;
ALTER TABLE tenants ADD COLUMN logo_url TEXT;
ALTER TABLE tenants ADD COLUMN invite_link_token TEXT UNIQUE;
ALTER TABLE tenants ADD COLUMN invite_link_enabled BOOLEAN NOT NULL DEFAULT FALSE;
```

`slug` is auto-generated from `name` on creation (lowercase, hyphens, unique suffix if collision). For existing rows, generated by migration.

### `organization_members`

```sql
CREATE TABLE organization_members (
  org_id     TEXT        NOT NULL REFERENCES tenants(id) ON DELETE CASCADE,
  user_id    TEXT        NOT NULL REFERENCES "user"(id) ON DELETE CASCADE,
  role       TEXT        NOT NULL DEFAULT 'member',  -- 'owner' | 'admin' | 'member'
  invited_by TEXT        REFERENCES "user"(id),
  joined_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  PRIMARY KEY (org_id, user_id)
);
CREATE INDEX organization_members_user_id_idx ON organization_members(user_id);
```

### `organization_invites`

```sql
CREATE TABLE organization_invites (
  id          UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id      TEXT        NOT NULL REFERENCES tenants(id) ON DELETE CASCADE,
  email       TEXT        NOT NULL,
  role        TEXT        NOT NULL DEFAULT 'member',
  token       TEXT        NOT NULL UNIQUE,
  invited_by  TEXT        NOT NULL REFERENCES "user"(id),
  expires_at  TIMESTAMPTZ NOT NULL,
  accepted_at TIMESTAMPTZ,
  created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
CREATE INDEX organization_invites_token_idx ON organization_invites(token);
```

Email invites expire after 7 days.

### Migration for existing data

For every row in `tenants`, insert one `organization_members` row: `(org_id=id, user_id=id, role='owner')`. Generate a slug from `name` (lowercase, replace spaces/special chars with hyphens, append 4-char suffix if collision).

## Auth Context & Org Resolution

`AuthUser` gains `org_id`:

```python
class AuthUser(BaseModel):
    id: str
    email: str
    org_id: str
```

**Resolution order in `get_current_user`:**

1. Read `X-Org-Id` header (forwarded by Next.js proxy from `active_org_id` cookie).
2. Validate user is a member of that org. If valid → use it.
3. Fallback: query `organization_members` for any org where `user_id = user.id AND role = 'owner'`. Use that org's id (personal org, where `org_id = user_id` for all existing users).
4. Guests: `org_id = None`. Guest budget remains keyed by IP as before.

**Downstream impact:** Every reference to `user.id` as the tenant/budget key switches to `user.org_id`. Since existing personal orgs have `org_id = user_id`, all existing data and behavior is preserved with zero migration of budget rows.

**Next.js proxy:** Reads `active_org_id` cookie and forwards as `X-Org-Id` header on all proxied requests to FastAPI.

## FastAPI API

Router: `apps/server/src/api/routers/organizations.py`

**Role enforcement:** `require_org_role(min_role)` dependency. Hierarchy: `member < admin < owner`.

### Org settings
| Method | Path | Auth | Description |
|--------|------|------|-------------|
| `GET` | `/organizations/current` | member+ | Org details + caller's role |
| `PATCH` | `/organizations/current` | admin+ | Update name, slug, logo_url |

### Members
| Method | Path | Auth | Description |
|--------|------|------|-------------|
| `GET` | `/organizations/current/members` | member+ | List all members |
| `PATCH` | `/organizations/current/members/{user_id}` | owner | Change member role |
| `DELETE` | `/organizations/current/members/{user_id}` | admin+ | Remove member (can't remove sole owner) |

### Email invites
| Method | Path | Auth | Description |
|--------|------|------|-------------|
| `POST` | `/organizations/current/invites` | admin+ | Create invite, send email |
| `GET` | `/organizations/current/invites` | admin+ | List pending invites |
| `DELETE` | `/organizations/current/invites/{invite_id}` | admin+ | Revoke invite |

### Shareable invite link
| Method | Path | Auth | Description |
|--------|------|------|-------------|
| `GET` | `/organizations/current/invite-link` | admin+ | Get token + enabled state |
| `PATCH` | `/organizations/current/invite-link` | admin+ | Enable/disable link |
| `POST` | `/organizations/current/invite-link/reset` | admin+ | Regenerate token |

### Accept invite (public)
| Method | Path | Auth | Description |
|--------|------|------|-------------|
| `GET` | `/join/{token}` | none | Validate token, return org name + role |
| `POST` | `/join/{token}` | required | Accept invite, insert member row |

`POST /join/{token}` checks: token exists, not expired, not already accepted (for email invites), user not already a member. For link invites, `invite_link_enabled` must be true.

## Web UI

### Org Switcher
Location: chat header, between `NotificationBell` and `UserMenu`.

A button showing the current org's initials/name. Opens a `Popover` listing all user's orgs (name, role badge). No "Create organization" option — additional org creation is out of scope; every user already has their personal org. Selecting an org sets `active_org_id` cookie and calls `router.refresh()`. Single user with one org sees no switcher (or a minimal display — no dropdown needed).

State: fetched once on mount from `GET /api/organizations/current/members` + a `GET /api/organizations` (list of user's orgs). Cached in component state, refreshed on org switch.

### `/settings/organization` tab
New tab in the existing settings page shell.

**General section** (admin+):
- Org name (text input)
- Slug (text input, shows preview: `app.example.com/org/slug`)
- Logo URL (text input with 40px preview)
- Save button

**Members section** (all members can view, admin+ can mutate):
- Table: avatar initials | name | email | role badge | actions
- Owner can change role via inline dropdown
- Admin/owner can remove members (Remove button → confirmation dialog)
- Cannot remove the sole owner

**Invitations section** (admin+):
- *Email invites:* email input + role `<select>` + "Send Invite" button. Below: list of pending invites (email, role, expires, Revoke button).
- *Invite link:* read-only URL input with Copy button + enable/disable `<Switch>` + "Reset link" button (confirmation required).

### `/join/[token]` page
Public route. Fetches `GET /api/join/[token]` on load.

- If valid: centered card — org name, role, "Accept Invitation" + "Decline" buttons.
- If already a member: redirect to `/chat`.
- If expired/invalid: error card with link to home.
- "Accept Invitation" requires auth — if unauthenticated, redirect to `/login?next=/join/[token]`.
- On success: set `active_org_id` cookie to new org, redirect to `/chat`.

### Next.js proxy routes
```
GET  /api/organizations/current           → FastAPI
PATCH /api/organizations/current          → FastAPI
GET  /api/organizations/current/members   → FastAPI
PATCH /api/organizations/current/members/[userId] → FastAPI
DELETE /api/organizations/current/members/[userId] → FastAPI
POST /api/organizations/current/invites   → FastAPI
GET  /api/organizations/current/invites   → FastAPI
DELETE /api/organizations/current/invites/[inviteId] → FastAPI
GET  /api/organizations/current/invite-link → FastAPI
PATCH /api/organizations/current/invite-link → FastAPI
POST /api/organizations/current/invite-link/reset → FastAPI
GET  /api/join/[token]                    → FastAPI
POST /api/join/[token]                    → FastAPI
```

The proxy reads the `active_org_id` cookie and appends `X-Org-Id` header to all requests.

## Signup Hook

`databaseHooks.user.create.after` in `packages/auth/src/index.ts` is extended to also:
1. Create a tenant row (`id = user_id, name = user.name ?? user.email, plan = 'free', token_limit = 100000`)
2. Generate a slug
3. Insert `organization_members` row `(org_id = user_id, user_id = user_id, role = 'owner')`

This is the primary creation path. `get_or_create_tenant` in `SessionRepository` remains as a safety-net upsert (called from `chat_service.py`) and must also be updated to insert the `organization_members` owner row if it doesn't exist.
